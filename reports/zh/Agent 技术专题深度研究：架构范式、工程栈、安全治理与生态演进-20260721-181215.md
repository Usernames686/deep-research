## 一、Agent 技术边界从对话助手转向可执行系统

> 界定 Agent 的核心内涵、能力边界与相邻概念，厘清其从聊天机器人、工作流自动化到自主执行系统的差异。 公开资料显示，本章应把 Agent 视为由模型、工具、状态与治理共同约束的工程系统，而不是单一模型能力的外溢；关键判断来自 [(1)](#ref1)、[(2)](#ref2)、[(3)](#ref3)。

### 1.1 概念界定

本节的核心判断是：概念界定 不能单独用模型参数或演示效果衡量，而要看它是否能在真实任务里形成可观测、可恢复、可审计的闭环。围绕问题“Agent 与聊天机器人、传统 RPA、工作流编排工具的本质差异是什么？”，可追溯事实显示，Lilian Weng 将 LLM-powered autonomous agents 拆解为 planning、memory、tool use 三类核心组件，仍是架构分析的经典框架。[(1)](#ref1) 这使 Agent 的技术边界从“回答问题”扩展到“调用外部系统并承担行动后果”。

从因果链看，Agent 的能力提升通常先表现为工具接口和任务状态的标准化，然后才转化为复杂任务完成率。OpenAI 在 2025 年推出 Responses API、内置工具、Agents SDK 和可观测追踪，标志 Agent 从对话接口转向可执行系统。[(2)](#ref2) 这意味着架构设计需要同时处理成功路径和失败路径：工具参数要可验证，执行日志要能回放，权限边界要能收窄，人工审批要能插入关键节点。
